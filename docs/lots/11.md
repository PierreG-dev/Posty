## 1. `compare-with-n8n.ts`

Trois modes, tous **read-only** (jamais de PATCH, jamais d'envoi) :

1. `--posty-side --from=YYYY-MM-DD --to=YYYY-MM-DD` : lit `mail_log`
   `dryRun=true` sur la fenêtre + `mail_queue` en `pending`, produit un JSON
   normalisé `{ companyId, sequenceStep, sentAt, source:'posty' }`.
2. `--twenty-side --from=... --to=...` : paginé `listCompanies({
   isAutoHandled: true })`, filtre `lastContactedAt` dans la fenêtre, déduit le
   step à partir de `followupCount - 1` (clampé à 2, comme n8n §1.2), produit
   un JSON `{ companyId, sequenceStep, sentAt, source:'twenty' }`.
3. `--compare --posty=... --twenty=...` : diff par `(companyId, sequenceStep)`.
   Quatre types d'écart classifiés : `missing_in_posty` (le pire cas — n8n
   envoyé, Posty non), `extra_in_posty` (Posty envoyé, n8n non),
   `step_mismatch` (même contact, step différent), `date_drift` (> 3 j
   d'écart sur une paire alignée). Verdict `green` (zéro écart) ou `red`.
   Code de sortie 2 sur `red` pour scripter la porte de bascule.

### Décisions

- **Fenêtre en Europe/Paris**, cohérent avec tout le reste de l'app.
- **Séquence uniquement** : les campagnes n'existent pas côté n8n (§5.3),
  hors comparaison.
- **La comparaison lit l'état actuel de Twenty**, pas des logs n8n exportés.
  Twenty *est* le journal de n8n (`lastContactedAt`, `followupCount`).
- **`step_mismatch` prime sur `missing_in_posty`** : si la clé
  `companyId#step` est absente côté Posty mais qu'un autre step existe pour ce
  contact, c'est un décalage — pas une absence. Sinon on double-comptait.

### Point ouvert

Le décalage entre `mail_log.sentAt` (Posty dryRun, instant simulé) et
`lastContactedAt` (n8n réel, timestamp Twenty) peut atteindre plusieurs heures
sur un même créneau : les deux ne partent pas exactement en même temps. Le
seuil `date_drift > 3 jours` est calibré large pour ça. À resserrer si on
observe des faux positifs après la première fenêtre.
