## Décisions

### 1. Registry visuel — interface d'abord
Le CDC §9.1 exige que le contrat, le validateur et le bouton « Copier le prompt » **dérivent** du registry. Lot 6 apporte les templates. J'ai introduit **dès maintenant** une interface `VisualRegistry` avec une impl stub (`emptyRegistry`) qui impose `visual: null` et `carousel: null`. Lot 6 remplacera `defaultRegistry()` sans toucher ni au générateur ni au validateur ni au prompt builder. Le point d'entrée est [src/modules/linkedin/visuals/registry.ts](../../src/modules/linkedin/visuals/registry.ts).

### 2. Post généré en mode `auto` → `status='draft'`, pas `queued`
Le CDC §7.2 dit « SINON persister (source='ai') puis publishPost(post) ». Il ne précise pas le statut intermédiaire. J'ai choisi **`draft`** pour deux raisons :
- La règle dure : « La file n'EST PAS consommée » (§7.2). Créer en `queued` polluerait `peekQueuedHead` pour le tick suivant si `publishPost` échouait avant `applyPublishOutcome`.
- `claimForPublishing` accepte `draft` → aucun impact fonctionnel.

### 3. `AutoFileSwitch` — pas de nouveau composant
Le plan prévoyait un composant dédié. Découvert à l'exploration : [src/modules/linkedin/ui/auto-switch.tsx](../../src/modules/linkedin/ui/auto-switch.tsx) existe déjà, monté sur `/linkedin`, patch `PATCH /api/linkedin/settings`. Rien à ajouter.

### 4. `ANTHROPIC_API_KEY` optionnel dans l'env
La règle projet dit « aucun secret en dur ». Mais imposer la clé au boot casse `typecheck` en CI sans clé. Compromis : `.optional()` dans le schéma env, vérification à l'exécution dans `RealAnthropicClient` (throw explicite avec message clair) et dans la route API (renvoie 503 avec le message si la clé manque).

### 5. Retry — 1 seul, avec l'erreur citée MOT POUR MOT
§8.8 exige « 1 retry avec l'erreur exacte citée au modèle ». Le message envoyé contient :
```
Ton JSON précédent a été rejeté par le validateur. Erreurs :
- content : Première ligne : 138 caractères > 100
- hashtags : hashtags: 6 fournis, attendu 3 à 5
Corrige et renvoie UNIQUEMENT le JSON valide.
```
Testé dans `generator.test.ts` : le 2e appel doit contenir « rejeté par le validateur ».

### 6. `PROMPT_VERSION`
Constante `v1-lot5` persistée dans `aiMeta.promptVersion`. À bumper à chaque changement de format qui change la voix — c'est ce qui permettra de repérer un « décrochage » qualité corrélé à un changement de prompt.

## Critère d'acceptation §18 CDC-01 — à valider en manuel

> « Je copie le prompt d'un thème, je le colle dans un chat, je récupère un JSON, je le colle dans l'app : il passe le validateur du premier coup. »

Procédure :
1. `/linkedin/themes/[id]` → « Copier le prompt complet ».
2. Coller dans un chat externe (Claude, ChatGPT, etc.), récupérer le JSON.
3. `/linkedin/posts/new` → onglet « Importer » → mode JSON → sélectionner le même thème → coller.
4. Attendu : `created: N`, aucun `error` dans la réponse de l'API.

À exécuter dès qu'une vraie `ANTHROPIC_API_KEY` sera renseignée.

## Points ouverts

- **Coût réel non mesuré** : le CDC estime ~1,4 ¢ par génération (§8.1). À confirmer sur les 10 premières générations réelles.
- **Génération de carrousels** : le mode `carousel` du thème est accepté par le validateur (via le registry stub qui impose `carousel: null` → un thème carrousel produira un warning cohérent mais aucun visuel). Lot 6 branche le vrai comportement.
- **Le champ `altText`** est capturé par le schéma mais aucun rendu visuel n'y correspond au lot 5. Utile dès lot 6.
- **Rate limits Anthropic** : pas de garde-fou explicite (retry sur 429, backoff). À revoir si le worker en auto tape trop souvent — improbable vu ~12 générations/mois.
