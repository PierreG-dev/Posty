# Lot 08 — Mailing : file d'envoi, boucle SMTP, PATCH Twenty

## Ce qui est livré

Le socle Mailing de l'étape 7 avait modèles, repos, client Twenty, service `greeting` et éditeurs UI mais **pas de file d'envoi ni de sender**. Ce lot ajoute la mécanique complète : éligibilité → enfilement idempotent → boucle d'envoi avec quota strict → PATCH Twenty après séquence. IMAP et campagnes suivent aux lots 9 et 10.

### Domaine

- `src/modules/mailing/domain/mail-queue.ts` — schéma zod complet : `kind`, `sequenceStep`, `campaignId`, `priority (1|2|3)`, `snapshot {name,email,greeting}`, `threading {inReplyTo,references}`, `status`, `attempts`, `lastError`, `cancelReason`.
- `src/modules/mailing/domain/mail-log.ts` — journal des envois RÉELS, source du quota quotidien.
- `src/modules/mailing/domain/campaigns.ts` — modèle minimal (nécessaire pour que `mail_queue.campaignId` puisse pointer une entrée valide et que l'index unique campagne soit exerçable dès aujourd'hui ; le compositeur arrive au lot 10).

### Repos

- `mail-queue-model.ts` — **les deux index uniques partiels** (§4.5), garde-fous anti-double-envoi :
  - `{ companyId, kind, sequenceStep }` unique, partial `{ kind:'sequence' }`
  - `{ companyId, campaignId }` unique, partial `{ kind:'campaign' }`
  - + tri `{ status, priority, createdAt }`.
- `mail-queue-repo.ts` — `enqueue` (duplicate silencieux sur E11000), `claimNextPending` (findOneAndUpdate atomique pending → sending), `markSent/Failed/Cancelled`, `retryFailed`, `cancelPendingForCompany`, `countPendingBreakdown`.
- `mail-log-model.ts` + `-repo.ts` — `logSent`, `countSentOnParisDay` (source de vérité du quota), `countBreakdownOnParisDay` (ventilation p1/p2/p3 pour le dashboard).

### Services

- `enqueue.ts` — `enqueueSequence(company, step)` : récupère template + blocs + salutation cached, appelle `renderSubject`/`renderTemplate`, fige `snapshot`, décide priority (`step 0 → 2`, `step 1|2 → 1`), threading (`step 0 → null`, sinon `{inReplyTo, references}` du fil précédent). Duplicate silencieux.
- `eligibility-tick.ts` — job quotidien : liste `isAutoHandled=true`, skip pausés / hard bounce / CLIENT / PARTENAIRE (§5.3), skip si `nextFollowupAt` dans le futur (comparaison en zone Paris), enfile `step = followupCount`.
- `smtp.ts` — interface `SmtpClient` isolée + `createNodemailerClient(settings)` (import dynamique) + `createDryRunClient()` (aucune socket, `messageId = dry-<uuid>@posty.local`).
- `twenty-patch-after-send.ts` — `buildAfterSendPatch` : construit le patch EXACT (`followupCount +1`, `lastContactedAt`, `lastMessageId`, `messageReferences` concaténées, `nextFollowupAt = sentAt + delays[step+1]` sinon absent, `toContact=false`). Aucun champ nouveau (§3.1).
- **`send-tick.ts`** — la boucle critique :
  - lock global `mail:send-tick` (TTL 30 min) ;
  - `settings.paused` → return immédiat ;
  - filtrage horaire via `isInSendWindow(settings, now)` (tolérance ±30 s, calcul en Paris via Luxon) ;
  - boucle : à chaque itération, **recompter le quota** dans `mail_log` du jour, `claimNextPending` (findOneAndUpdate atomique), re-vérification tardive `getMeta` (paused/bounce → cancelled sans consommer le quota), appel SMTP (real ou dryRun selon `settings.dryRun`), `logSent`, `markSent`, puis PATCH Twenty **uniquement pour les séquences hors dryRun**, puis `sleep(jitter)` ;
  - échec SMTP → `markFailed` définitif + Pushover (pas de retry auto, décision plan).
- `send-tick.ts` expose aussi `runSendLoop({ smtp, twenty, now })` — bypasse la fenêtre horaire, utilisé par les tests.

## Décisions prises (rappel des points ambigus levés en début de lot)

1. **Retry SMTP** : `failed` définitif + Pushover + bouton « Réessayer » manuel. Un incident SMTP ne re-brûle pas 3 envois automatiquement. Le quota du jour n'est pas consommé sur échec.
2. **`paused` / `dryRun` & éligibilité** : l'éligibilité tourne QUAND MÊME. Les deux drapeaux s'expriment uniquement à l'envoi. Ça permet de lever `paused` sans attendre 24 h, et surtout de recetter en dryRun (sinon rien n'est en file).
3. **Déclenchement `send-tick`** : tick minute + `isInSendWindow ± 30 s`. Reload des créneaux à chaud sans redémarrer le worker, lock global empêche double exécution.

## Ce qui reste ouvert / à surveiller

- **Dépendance ajoutée** : `nodemailer@^9.0.3` + `@types/nodemailer`. Import dynamique dans `smtp.ts` — le module n'est chargé qu'à la 1re connexion réelle.
- **Threading de séquence** : les tests couvrent la présence des headers `In-Reply-To` / `References`, mais la validité du `messageReferences` initial (posé par n8n aujourd'hui) devra être vérifiée en dryRun grandeur nature (lot 11).
- **Balayage Twenty en éligibilité** : `listCompanies({ isAutoHandled: true, limit: 500 })` — au-delà de 500 contacts, la pagination n'est PAS gérée. À revoir si le nombre de contacts éligibles explose.
- **Jitter en tests** : forcé à 0. En prod, le `sleep(45–180 s)` allonge un cycle d'envoi de 25 mails à ~15–75 min, ce qui est le comportement voulu.
- **`sentAt` du log = horloge réelle** au moment de l'appel SMTP, pas le `now` passé au tick. C'est intentionnel (le journal doit refléter l'instant SMTP), mais rend les tests un peu plus subtils : ils comparent `nextFollowupAt` à `sentLog[0].sentAt` et non au `now` passé.
- **Boucle SMTP crash-safe** : si le worker crashe entre `logSent` et `markSent`, l'index unique sur `messageId` empêche le doublon en re-log. Le status côté `mail_queue` restera à `sending` — le lock TTL (30 min) libèrera le sender, mais l'entrée n'est pas re-claim (status = `sending` ≠ `pending`). C'est un choix : mieux vaut une entrée bloquée en `sending` (visible à l'UI, réparable à la main) qu'un doublon.
- **Pas encore branché en prod** : recette dryRun requise avant `paused=false && dryRun=false` (cf. lot 11).
