# Lot 04 — Scheduler LinkedIn

> Étape la plus dangereuse : une erreur ici publie deux fois, ou publie à 3 h
> du matin. Ce lot ajoute les créneaux, le worker qui tire dessus, les deux
> garde-fous d'idempotence, la vue calendrier et le dashboard v1.

### Scheduler (le cœur)
- `src/modules/linkedin/services/scheduler.ts` — **fonctions pures**,
  testables sans DB :
  - `computeDueSlots(slots, now)` — renvoie les slots dus, ancrés Paris via
    Luxon, dans la fenêtre [scheduled, scheduled + 15 min].
  - `computeDueOneShots(posts, now)` — pareil pour les `scheduled`, avec un
    flag `missed` si `now - scheduledAt > 15 min`.
  - `slotIdempotencyKey`, `slotLockKey`, `oneShotIdempotencyKey` — clés
    stables ancrées jour/minute Paris.
- `src/modules/linkedin/services/scheduler-tick.ts` :
  - `resolvePublication(due)` — implémente §7.2 à la lettre :
    - mode effectif = `slot.modeOverride ?? (settings.autoGeneration ? 'auto' : 'queue')`
    - `queue` vide → `publications.empty_queue` + Pushover 🚨, PAS de repli
      thème (choix explicite CDC).
    - `queue` non vide → `publishPost(head, mode:'queue', slotId, idempotencyKey)`.
    - `auto` → écrit `publications.skipped` avec `error: "auto mode not
      implemented (lot 5)"`, log `warn`, PAS de Pushover, la file N'EST PAS
      consommée. Le point d'extension est prêt pour brancher `generatePost`
      au lot 5 (une seule ligne à ajouter).
  - `runSchedulerTick(now)` — orchestre : liste slots + one-shots dus,
    traite chaque slot sous verrou par-slot-par-jour, traite les one-shots
    manqués/dus.

### Idempotence — deux garde-fous, tous deux testés
1. Verrou Mongo `publish:{slotId}:{YYYY-MM-DD}` via `withLock` (TTL 5 min).
   Autorise le parallélisme entre slots simultanés mais unicité par
   slot/jour. Un tick concurrent qui ne prend pas le verrou log
   `slot_locked` (debug) et sort proprement.
2. Index UNIQUE sur `publications.idempotencyKey` (déjà en place). Le
   scheduler reconnaît `duplicate` et sort silencieusement — ni notif, ni
   erreur.

## Décisions et réponses aux 3 questions du plan

1. **Rattrapage one-shots.** Même règle que les créneaux récurrents : fenêtre
   `CATCHUP_WINDOW_MIN = 15 min`. Au-delà, le post repasse en `status:
   'failed'` avec `lastError` explicite (« Créneau one-shot manqué (…)»),
   une entrée `publications.skipped` est écrite (idempotencyKey =
   `oneshot-{postId}`, ne se répète pas), et Pushover priorité 0 avec lien
   vers la fiche. L'UI proposera « replanifier » — l'action existe déjà
   (édition de `scheduledAt`).

2. **Mode `auto` sans `generatePost`.** Décision : `logger.warn` +
   `publications.skipped` (pas `failed`), PAS de Pushover. Motif : le mode
   n'est pas activé par défaut ; s'il est activé volontairement avant le
   lot 5, l'utilisateur sait pourquoi le silence. Le point d'extension est
   explicite dans `scheduler-tick.ts` — au lot 5, il suffit d'importer
   `generatePost` et de remplacer le bloc.

3. **Compte à rebours dashboard.** Respecte `modeOverride` — la projection
   `projectUpcoming` calcule le mode effectif par slot, et le dashboard
   affiche le badge IA/File du prochain.

## Tests

`tests/scheduler.test.ts` — 14 tests **purs** (aucun mock DB) :
- ancrage Paris malgré `process.env.TZ='UTC'`
- fenêtre de rattrapage 15 min (inclusive à 15, exclusive à 16)
- créneau raté de 3 h → non dû
- slot inactif / mauvais jour → non dû
- one-shots : dus / manqués / status ignoré / futur ignoré
- clés d'idempotence et de verrou

`tests/scheduler-tick.test.ts` — 8 tests **d'intégration mockée** :
- file vide → `empty_queue` + Pushover, `publishPost` non appelé
- file non vide → `publishPost` avec idempotencyKey ancrée Paris/minute
- `modeOverride='queue'` écrase `autoGeneration=true` global
- mode auto → `skipped`, file préservée, pas de Pushover
- deux ticks concurrents → 1 seul `publishPost` (verrou)
- deux ticks séquentiels en mode auto → 1 seule publication (index unique)
- one-shot manqué de 3 h → `skipped` + `applyPublishOutcome(failed)` + notify
- one-shot dans la fenêtre → `publishPost(oneshot-key)`

Résultat : **12 fichiers, 86 tests verts**.

  le bloc `auto_not_implemented`), retirer le log warn.
- **UI « replanifier »** pour un one-shot repassé en `failed` — le champ

  mais l'action n'a pas de bouton dédié aujourd'hui. Non-bloquant.
- **Alerte « file sous seuil »** : affichée dans l'UI (§12) mais pas encore
  envoyée via Pushover. Le §11 la liste comme priorité 0 ; à ajouter avec
  un job quotidien léger, ou à l'issue d'un `publishPost` qui vide la
  file. Non demandé explicitement au lot 4, à faire au lot 5 ou dans un
  petit lot d'affinage.
- **Test avec vrai Mongo en mémoire** : les tests actuels mockent
