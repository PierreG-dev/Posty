## Décisions et pourquoi

### npm (pas pnpm) — sur ta demande.

### ESLint `no-restricted-imports` seul (pas de test Vitest en doublon)
Le CDC accepte l'un OU l'autre. La règle ESLint donne le feedback en IDE et bloque le CI/pre-commit ; un test Vitest redondant n'apporterait rien.

### argon2id + hash **wrappé base64** dans `AUTH_PASSWORD_HASH`
Le format PHC natif (`$argon2id$v=19$...`) contient des `$` qui cassent l'interpolation shell et docker-compose (`${VAR}`). Je stocke `Buffer.from(phc).toString('base64')` en env ; `password.ts` décode avant `argon2.verify()`. `scripts/hash-password.ts` fait le travail pour toi.

### JWT HS256 via `jose` pour la session
Standard, léger, compatible Edge runtime (le middleware Next tourne en Edge). Alternative écartée : `iron-session` (cookie chiffré symétrique) — moins standard, pas d'avantage ici.

### `tsc-alias` pour le build worker
Ajout **non listé dans le plan initial**. Raison : `tsc` ne réécrit pas les alias `@/*` du `tsconfig.paths` dans la sortie JS, donc `node dist-worker/worker/index.js` échouait avec `Cannot find module '@/modules/shared/...'`. `tsc-alias` fait la réécriture post-compile en imports relatifs. Alternative envisagée : `tsx` en prod (plus lourd, ajoute une dep de runtime).

### Rate-limit login in-memory (pas Mongo)
Documenté dans CLAUDE.md. Mono-utilisateur, une seule instance web côté Coolify : suffisant. Si un jour on multi-replica le web, la migration vers Mongo prend 20 lignes.

### `_next/static` et fichiers statiques exclus du middleware
Matcher : `/((?!_next/static|_next/image|favicon.ico|.*\..*).*)`. Sinon chaque asset paie un round-trip de vérification cookie.

## Points ouverts / à ne pas oublier

1. **Aucun secret n'est en dur nulle part**. `.env.example` est prêt ; l'utilisateur doit générer localement `AUTH_PASSWORD_HASH`, `SESSION_SECRET`, `ENCRYPTION_KEY`. Le README explique comment.
2. **Aucune connexion réelle** à Mongo/LinkedIn/Anthropic/SMTP/IMAP n'a été tentée — c'est le rôle des lots suivants. La connexion Mongo a été vérifiée pendant les tests runtime du worker (Mongo local répondait sur `localhost:27017`).
3. Le squelette de `/settings` est en lecture seule. L'édition (dryRun, autoGeneration, seuils, Pushover, LinkedIn) arrivera aux lots 3–5.
4. Le middleware exempte déjà `/api/linkedin/callback` — utile dès le lot 3.
5. **Attention Windows** : le hash argon2 embarque des bindings natifs. En dev sous Windows, `npm install` a compilé sans souci (Node 22.19). Pour Docker, l'étape `deps` installe `python3 / make / g++` via alpine — testé au moins pour `npm ci --include=dev`.
6. **Prochaine étape (spike LinkedIn — lot 1)** : contexte propre recommandé (`/clear`). Le mode d'emploi le rappelait ; tu as choisi d'enchaîner, note prise.
