## Garde-fous du script

- **`--dry` par défaut** : sans `--live` explicite, aucun appel HTTP réel n'est émis. Les payloads sont juste affichés. C'est ce qui évite de publier accidentellement 26 posts de test sur ton profil.
- **`--cleanup`** : `DELETE /rest/posts/{urn}` sur tous les URN créés pendant le run. Ce sont les seuls posts que le script a créés, tracés dans un tableau local.
- **`--only`** et **`--chars`** : réduire le périmètre pour itérer vite (ex : `--only=chars --chars="@ |"`).
- **`--delay-ms`** : espacement configurable entre deux appels HTTP dans le test de chars, pour ne pas taper au rate limit (`~100/jour/membre` — le run complet en fait 27 au maximum, très en dessous).
- Une exception est catchée en haut de main : `process.exit(1)` avec message court, pas de stack trace envahissante.
- **La synthèse est écrite dans un `finally`** : même si un test explose, on récupère la trace des tests passés + les URN à nettoyer manuellement.

## Décisions et pourquoi

### Aucune dépendance sur l'app (`src/`)
Le CDC dit « script jetable ». Le spike doit tourner sur une checkout vierge sans avoir à démarrer Mongo. Il utilise `fetch` natif et Node natif uniquement. `tsx` suffit à le lancer.

### Le PNG et le PDF sont **inlinés**, pas lus depuis le disque
Un blob PNG 4×4 rouge en base64 (~70 octets) et un PDF minimal 1-page (~350 octets) sont construits inline. Zéro fichier à gérer, zéro chemin à passer, zéro dépendance à `pdf-lib` (qu'on installera au lot 6 seulement).

### Le test 5 (commentaire) réutilise un URN du test 1
Si le test 1 a réussi, on commente son post. Sinon on publie un post de support à la volée. Ça évite de dépendre d'un URN externe et rend le test 5 exécutable en isolation via `--only=comment`.

### Le résultat est aussi sérialisé en JSON
`docs/spike-linkedin.json` — permet de coller la sortie brute dans un chat plutôt que de retranscrire à la main. Le fichier est gitignoré parce qu'il contient des URN pouvant identifier ton profil.

### `--dry` est le défaut, pas `--live`
Un typo qui déclenche 26 publications réelles est le pire scénario possible. On inverse la responsabilité : il faut **explicitement** demander `--live` pour toucher LinkedIn.

## Exécution live — résultats

Run effectué le **2026-07-14 14:14 UTC**, API `202506`, 25 posts créés puis tous supprimés par `--cleanup` (0 fuite). Rapport détaillé : [docs/spike-linkedin.md](../spike-linkedin.md).

**Verdicts finaux** :

| Test | Résultat | Impact |
|---|---|---|
| 1 — Publication texte `/rest/posts` | ✅ 201 | Le lot 3 peut abandonner `/v2/ugcPosts` (déprécié, utilisé par n8n actuel). |
| 2 — Échappement `commentary` | ⚠️ Ambigu | Décision prudente : **échapper les 13 caractères** au lot 3. Voir Test 2 du rapport pour la nuance (les 4 échecs 422 étaient des « duplicate content », pas des « invalid character »). |
| 3 — Publication avec image | ✅ 201 | Pipeline en 3 étapes validé pour le lot 3. |
| 4 — Publication avec PDF (Documents API) | ✅ 201 **(bonne surprise)** | **Carrousel PDF FAISABLE** au lot 6. Le scope `w_member_social` suffit. |
| 5 — Premier commentaire (Social Actions) | ❌ 403 `ACCESS_DENIED partnerApiSocialActions` | **PAS FAISABLE**. Repli propre : stocker + copier à la main + rappel Pushover après chaque publication. |

**Point technique à ne pas oublier au lot 3** : les posts document reviennent en URN `urn:li:ugcPost:...` (et non `urn:li:share:...` comme les posts texte/image). Le client `LinkedInClient` et le stockage `posts.linkedin.urn` doivent accepter les deux formes. Le `DELETE /rest/posts/{urn}` a fonctionné pour les deux.

## Le spike a fait son travail

Les trois incertitudes sont levées, y compris celle qui a eu une réponse contraire à celle du CDC (Documents API accessible → carrousel possible). Les lots 3 et 6 sont débloqués :

- **Lot 3** — client LinkedIn + `escapeCommentary()` (13 caractères, un test unitaire chacun) + repli premier commentaire (`firstComment.status` réduit à `none | pending`, bouton « Copier » + Pushover).
- **Lot 6** — registry de templates avec `kind: 'both'` autorisé (post + slide), pipeline carrousel PDF actif, l'URN parseur tolère `urn:li:ugcPost:` et `urn:li:share:`.

**Étapes suivantes possibles** : lots 2 (thèmes/posts/file) OU lot 7 (mailing) — ils sont indépendants entre eux et du spike. Le CDC recommande de finir LinkedIn d'abord (2→3→4→5→6) avant d'attaquer Mailing.

**Attention** — la première fois que tu lances `--live`, garde un œil sur ton profil LinkedIn dans un autre onglet. Si l'un des posts a fuité (échec du `--cleanup` sur un URN), tu le vois passer et tu peux le supprimer à la main. Le script log chaque `DELETE ...` en fin de run.
