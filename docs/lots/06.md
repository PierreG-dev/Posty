## Décisions

### 1. Templates en `.ts` avec helper `h()`, pas en `.tsx`
Satori accepte des POJO `{ type, props, key }` — aucune dépendance à React. J'ai écrit un `h(type, props, ...children)` (17 lignes) et les templates sont des `.ts`. Bénéfices :
- Vitest n'a besoin d'aucune config JSX (pas de `@vitejs/plugin-react` pour les tests visuels).
- Aucune fuite de React dans le module `visuals/` — cohérent avec la règle projet « aucun import inutile ».
- Le typage `render(params, tokens): ReactElement` est conservé pour l'API du registry.

### 2. Contract → union discriminante, PAS `discriminatedUnion`
Le contrat §8.7 met le discriminant `templateId` **à l'intérieur** de l'objet `{ templateId, params }`, donc `z.discriminatedUnion` (qui exige un discriminant au niveau racine) ne s'applique pas. J'utilise une union classique `z.union([...])` — les erreurs sont un peu moins précises mais le validateur atteint quand même la bonne branche via le champ literal `templateId`.

### 3. Auto-enregistrement des templates — pas de side-effect depuis registry.ts
J'avais d'abord placé `import "./register";` à la fin de `registry.ts` pour rendre `defaultRegistry()` autoportante. Résultat : en ESM/vitest, le cycle `registry.ts ↔ register.ts ↔ templates/*` casse l'ordre d'export — `registerTemplate` était `undefined` au moment où les templates s'exécutaient (tests rouges avec `TypeError: registerTemplate is not a function`).

**Correction** : les 4 consommateurs de `defaultRegistry()` (validate-generated-post, prompt-builder, render, route templates) importent explicitement `@/modules/linkedin/visuals/register`. Un side-effect import par entrée, sans cycle. `defaultRegistry()` reste une simple fonction qui renvoie `realRegistry`.

### 4. `params` de slide typé `unknown` — cast au parse
Les routes valident les slides avec `z.object({ templateId: z.string(), params: z.unknown() })`. Zod traite `z.unknown()` comme optional par défaut, d'où `params?: unknown` dans le type inféré, incompatible avec `SlideParams` (params requis). Correction faite au call-site : `s.params ?? null`. Ça reste un compromis — l'alternative propre serait une refonte du typage de `SlideParams` pour accepter `undefined`, mais ce n'est pas justifié par un cas réel.

### 5. dryRun rend les visuels (choix tranché)
Le générateur produit déjà le PNG/PDF **au moment de la création du post** (avant même la mise en file), donc en dryRun le publisher n'a rien à re-rendre : l'asset est déjà en base. Le `payloadSnapshot` archivé embarque simplement `dryRunAssetId` pour permettre de retrouver le rendu via `/api/assets/[id]` pendant la recette. Zéro appel HTTP LinkedIn, mais le journal `publications` contient l'empreinte visuelle réelle.

### 6. Le formulaire params est un textarea JSON, pas un formulaire dynamique par template
J'ai considéré générer un formulaire à partir du schema zod de chaque template. Coût : ~200 lignes supplémentaires, plusieurs cas particuliers (textarea code, arrays). Bénéfice réel : marginal — le vrai flux est **génération IA → aperçu → publication**, pas édition à la main. Le textarea JSON couvre le tuning ponctuel et laisse la place à un vrai formulaire si l'usage le justifie.

### 7. Un template = 1200×1200 fixe
Le CDC §9.3 impose ce format (« le plus performant en feed »). Pas de paramétrage — c'est une constante `CANVAS_SIZE` exportée par `render.ts`. Si un jour on veut du portrait/paysage : nouveau lot, nouvelle constante.

## Points ouverts

- **Le carrousel n'a pas été testé bout en bout en dryRun.** Le pipeline generator→render→saveGeneratedPdf→publisher.dryRun est couvert par des tests unitaires mais pas par un e2e. À faire à la première génération IA carrousel réelle.
- **Rendu de blocs multi-lignes dans les templates.** Satori a un support flex/flexbox partiel : dans certains templates (checklist, before-after), un texte très proche des limites peut légèrement déborder. Les schemas laissent un peu de marge (65 col pour les puces, 40 pour les colonnes de code) — à ajuster à l'usage.
- **`params` en `Schema.Types.Mixed` dans MongoDB** — pas d'introspection possible côté DB. Suffisant pour l'usage actuel (retrouver et régénérer un asset), mais si un jour on veut faire des stats « quel template rend le mieux », il faudra durcir.
- **Polices commit dans le repo (~800 Ko).** OK pour maintenant, à ré-évaluer si le repo dépasse 100 Mo.
- **Le UI éditeur ne prévisualise pas le mode carousel.** Un thème `mode=carousel` monte `VisualEditor` en mode `carousel` — mais l'UI attend `mode=image` (POST preview kind=image). Le carrousel se génère à l'usage via l'IA (flux le plus fréquent), la retouche manuelle d'un carrousel viendra si le besoin apparaît.
- **Test dryRun avec vrai asset** manquant : le test publisher existant utilise `media.kind='none'`. À ajouter au lot 11 (recette parallèle).
