# Lot 09 — Mailing : IMAP (archivage, bounces, réponses)

## Ce qui est livré

Le lot 08 avait la boucle d'envoi complète mais aucun retour d'information
côté messagerie : pas d'archivage, aucune détection de bounce, aucune
détection de réponse. Ce lot ferme la boucle : le mail parti est archivé
dans un dossier « Posty », les rapports de non-remise sortent
automatiquement les adresses mortes, et une réponse humaine met le contact
en pause avec alerte Pushover — sans jamais changer son statut Twenty
automatiquement (§8.2, critère non négociable).

### Domaine et repos

- `src/modules/mailing/domain/mail-imap-state.ts` — watermark UID par
  dossier IMAP scanné. Réponse utilisateur au plan §9 : **watermark UID
  uniquement**, pas de `\Seen`, pas de move, l'INBOX reste intacte.
- `src/modules/mailing/repositories/mail-imap-state-{model,repo}.ts` — un
  singleton `mail_imap_state`, `folders[name] = { uidValidity, lastUid }`.
  `reconcileFolder(folder, serverUidValidity)` reset le watermark si
  l'INBOX a été recréée (UIDVALIDITY changée) — mieux vaut re-scanner que
  rater un bounce.
- Extension `mail_settings.imap` : `inboxFolder` (défaut "INBOX") et
  `spamFolder` (défaut "Spam"). Réponse utilisateur au plan §9 :
  **INBOX + Spam scannés tous les deux**, §15 point ouvert 4 clos.
- Extension `TwentyCompanyPatch` : le champ `status` (existant côté Twenty,
  jamais écrit par n8n) devient patchable par Posty via la seule action
  « Passer en CLIENT » — §3.1 respecté, aucun champ nouveau.

### Services

- **`imap.ts`** — client isolé et mockable, 4 méthodes (`ensureFolder`,
  `append`, `fetchNewMessages`, `close`). Impl `createImapflowClient` avec
  `imapflow` chargé dynamiquement (même pattern que `nodemailer` — pas de
  socket ouverte en test unitaire). Parseur MIME minimal `parseRawMime`
  exporté pour usage direct dans les tests et pour le scan.
- **`mime-build.ts`** — construction d'une trame RFC 5322 pour l'APPEND
  IMAP. Volontairement pas nodemailer : la trame est simple (headers +
  body texte brut UTF-8), et le contrat critique est qu'elle soit
  ré-parsable par notre propre parseur (testé).
- **`dsn-parse.ts`** — détection `multipart/report;
  report-type=delivery-status` puis extraction `Final-Recipient` + `Status`
  (`5.x.x → hard`, `4.x.x → soft`). Optionnel : `Diagnostic-Code` pour
  contexte.
- **`imap-inspect-tick.ts`** — la boucle §8. Une seule connexion, deux
  dossiers (INBOX + Spam), deux traitements séquentiels par message :
  d'abord DSN (bounces), sinon réponse. Watermark UID appliqué par dossier,
  `msg.from === settings.smtp.from` filtre l'auto-loop.
  - Hard bounce (`5.x.x`) → `setBounce(hard)` + `Twenty.PATCH { isAutoHandled: false }`
    + `cancelPendingForCompany` + Pushover priorité 1.
  - Soft bounce (`4.x.x`) → compteur incrémenté ; **3 soft consécutifs →
    escalade en hard** (même traitement complet).
  - Réponse (résolue par `In-Reply-To`/`References` → `mail_log.messageId`,
    ou à défaut `From` → `findLastLogByEmail`) → `setPaused(reply)` +
    `cancelPendingForCompany` + Pushover, **jamais de changement de statut
    Twenty**. Idempotent : si `paused && pausedReason==='reply'` déjà, on
    ne re-notifie pas.

### Modification de `send-tick.ts`

- Nouvelle dep injection `imap?: ImapClient | null`. Défaut :
  `null` en dryRun ou si `settings.imap.host` vide ; sinon
  `createImapflowClient(settings)` avec fallback silencieux en `null` sur
  échec de connexion (l'envoi doit pouvoir tourner même IMAP down).
- Après `logSent`/`markSent`, on tente `imap.append(folder='Posty', mime,
  flags=['\\Seen'])`. **Critère d'acceptation §7.2 :** un throw ici ne
  renvoie JAMAIS. `markLogImapArchived(logId, false)` + Pushover priorité 0
  + on continue vers le PATCH Twenty. Le mail EST parti.
- Nouveau champ `mail_log.imapArchived` peuplé (auparavant toujours faux).

## Décisions prises (rappel des points ambigus levés en début de lot)

1. **Gestion des messages scannés** — watermark UID uniquement (réponse
   utilisateur), pas de `\Seen`, pas de move. L'INBOX reste intacte côté
   client mail. Contrepartie : si le singleton `mail_imap_state` est
   supprimé, on re-scanne tout ; parade : re-notif filtrée par
   `paused && reason==='reply'` déjà en place et `bounce.kind` déjà posé,
   donc idempotent en pratique.
2. **INBOX + Spam scannés** (réponse utilisateur). Nom du dossier spam
   configurable via `settings.imap.spamFolder`. Un dossier absent = log
   warn silencieux, on continue avec l'autre.
3. **Cron IMAP à 7 h Paris** — 1 h après l'éligibilité (6 h), 3 h avant
   le premier créneau d'envoi (mar 10:30). Les deux jobs ne se croisent
   pas ; les entrées annulées à cause d'un bounce ou d'une réponse ne
   partent donc jamais dans la même journée.
4. **Résolution d'un bounce → contact** — via `mail_log.toEmail` (dernier
   envoi Posty vers cette adresse), pas via Twenty. Ça permet de tracer
   un bounce même si le contact a été supprimé côté Twenty entre-temps
   (rare, mais possible).
5. **Résolution d'une réponse** — priorité au threading (In-Reply-To /
   References contenant un `mail_log.messageId`), puis fallback sur
   `From` → `findLastLogByEmail`. Le fallback couvre les clients mail
   qui perdent le threading.
6. **Auto-loop** — un message dont `From` = `settings.smtp.from` est
   ignoré (empêche que Posty ne se prenne lui-même pour un contact quand
   le BCC `logs@` revient via l'INBOX pendant la phase de bascule).

## Ce qui reste ouvert / à surveiller

- **Dépendance ajoutée** : `imapflow@^1.4.7`. Import dynamique dans
  `imap.ts` — jamais chargée côté vitest ni côté build web.
- **Compteur soft bounce non remis à zéro sur envoi réussi** — décision de
  simplicité : si un contact soft-bounce 2 fois puis reçoit un mail OK
  puis re-soft-bounce une seule fois, il sera traité comme hard. En
  pratique, un contact qui soft-bounce plusieurs fois est déjà suspect.
  À ajuster si les faux positifs deviennent visibles.
- **Sujet DSN pas restreint** — on prend tout `multipart/report` avec
  `delivery-status`, même si le sujet n'est pas « Undelivered mail ». Ça
  attrape les DSN atypiques (postmaster.local, etc.) mais peut, en
  théorie, mal réagir à un rapport MDN (message-disposition-notification)
  mal typé. Non observé jusqu'ici.
- **Champ `status` Twenty écrit par la promotion** — si le champ n'existe
  pas sous ce nom exact côté Twenty, le PATCH échouera avec une 400 et
  l'action inbox affichera l'erreur. Ajuster le nom dans le patch de
  `/api/mailing/inbox/[id]/promote` le cas échéant. C'est le seul endroit
  du code qui écrit ce champ.
- **Pas encore branché en prod** : recette dryRun requise avant
  `paused=false && dryRun=false`, et un premier envoi manuel vers une
  adresse morte + une adresse répondante pour valider le cycle complet
  en réel (cf. lot 11).
