## Décisions et pourquoi

### Seuil de troncature « …voir plus » = 140 caractères
Le CDC ne le donne pas. LinkedIn coupe autour de cette valeur en feed desktop, moins sur mobile. Choix prudent et documenté dans `post-preview.ts`. La constante `LINKEDIN_TRUNCATE_AT` est exportée : si l'observation en prod montre un autre seuil, un seul point à changer.

### `sourceExternalId` ajouté au modèle `posts`
Pas au §6.3 mais indispensable pour l'idempotence §16 (« ne réimporte pas un `id` déjà présent »). Index **unique sparse** : les posts non issus de la migration ont `null` sans jamais collisionner. Écrit **exclusivement** par `scripts/migrate-from-sheets.ts` — jamais exposé dans les formulaires ni les routes de création manuelle.

### Composants UI en Tailwind pur (pas shadcn)
Cohérent avec le lot 0. Petits primitives typés dans `shared/ui/primitives.tsx`, tous branchés sur les tokens (`bg`, `surface`, `accent`, `queued`, `published`, `failed`…). Zéro dépendance shadcn ajoutée. Le fichier `components.json` reste posé (lot 0) au cas où un futur composant complexe justifierait un scaffold.

### Drag & drop = @dnd-kit (une seule dep supplémentaire)
`@dnd-kit/core` + `@dnd-kit/sortable` + `@dnd-kit/utilities`. Accessibilité clavier native. Pas de `react-beautiful-dnd` (non maintenu).

### DnD actif seulement quand un thème est filtré
Sans filtre thème, réordonner mélangerait les files de tous les thèmes — sans intérêt. L'UI affiche un hint : « 💡 Sélectionne un thème dans le filtre pour activer le drag & drop ».

### Garde-fou du reorder = les IDs doivent couvrir EXACTEMENT la file
`reorderQueue(themeId, orderedIds)` fait un `find` puis compare les deux sets. Si un ID est absent ou en trop → refus `409` sans écriture, message : *« orderedIds contient X ids, la file en a Y »*. Empêche qu'un client mal synchronisé (autre onglet, cache) écrase des positions par erreur.

### Onglet « Générer » visible mais désactivé
Rappel visuel qu'il arrive au lot 5. Meilleur UX qu'un onglet manquant qui pousserait à contourner la roadmap.

### Champ `firstComment` : `pending` par défaut si texte présent
Le spike (lot 1) a montré que l'API commentaire est inaccessible avec `w_member_social`. Toute saisie d'un premier commentaire passe `firstComment.status: 'pending'` = à coller à la main. Le workflow est expliqué dans le hint du champ.

### `dryRun` = `true` par défaut sur le singleton settings
Le lot 0 le posait déjà. Cohérent avec la philosophie « safe by default ».

### Règle ESLint `react/no-unescaped-entities` désactivée
Bruit pénible en français (« l'aperçu », « d'un », « qu'on »…). N'apporte aucune sécurité, seulement du travail.

## Critère de fin (prompt étape 2)

> Je crée un thème, je colle 10 posts séparés par `---`, je les affecte au thème, ils entrent dans la file dans le bon ordre, je les réordonne en drag & drop et l'ordre survit à un rechargement.

- ✅ Création thème (UI et API).
- ✅ Collage de 10 posts séparés par `---` (validé end-to-end en runtime).
- ✅ Affectation au thème (via `themeId` du lot ou via bulk-assign-theme après coup).
- ✅ Ordre 0..9 respecté à l'insertion.
- ✅ Reorder via `POST /posts/reorder` (drag & drop UI + API testée end-to-end).
- ✅ Persistance après rechargement (validé par re-`GET`).

## Ce qui reste ouvert / à ne pas oublier au lot 3

1. **`escapeCommentary()`** — le lot 3 doit l'implémenter avec les 13 caractères du spike, un test unitaire par caractère. Le rappel figure dans `docs/spike-linkedin.md > Décisions`.
2. **URN parseur** — le lot 3 doit accepter `urn:li:share:...` ET `urn:li:ugcPost:...` (documents PDF).
3. **Premier commentaire** — `firstComment.status` reste `none | pending` tant que l'API n'est pas ouverte. Ne pas ajouter `posted`/`failed` : ils n'existent pas dans le repli actuel.
4. **Migration** — j'attends le CSV exporté du Sheets pour rejouer le script en `--live`. Le workflow post-migration est documenté dans le script lui-même.
5. **Assets** — le champ `media.assetId` référence une collection `assets` qui viendra au lot 6 (Visuels). Aucun post à ce lot 2 n'a de média non-null.
